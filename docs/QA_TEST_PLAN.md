# 校园二手集市测试计划

## 1. 文档说明

本文档由测试与 QA 负责人编写，面向课程项目“校园二手物品交易 Web 平台”的测试设计、执行和验收。当前仓库实际技术栈为 Spring Boot 后端、Vue/Vite 前端、H2/MySQL 可切换数据库，因此测试方案按现有代码实现制定。

## 2. 测试目标

测试目标是验证系统能够稳定支撑“发布商品、浏览搜索、私信沟通、预定商品、完成交易”的校园二手交易闭环，并重点检查状态机、权限控制、异常输入和并发预定场景。

主要目标如下：

| 目标 | 说明 |
|---|---|
| 功能正确性 | 核心接口和页面流程符合需求说明 |
| 状态一致性 | 商品状态和订单状态按规则流转 |
| 权限安全 | 非本人、未登录、非参与者不能越权操作 |
| 异常兜底 | 非法输入、重复操作、已下架商品等场景有明确错误响应 |
| 并发可靠性 | 多买家同时预定同一商品时只能成功一个 |
| 性能可接受 | 商品列表、搜索、详情和预定接口在课程演示规模下响应稳定 |
| 可维护性 | 测试文档、用户手册和维护说明可支撑后续迭代 |

## 3. 测试范围

### 3.1 本轮覆盖范围

| 模块 | 覆盖内容 |

| 收藏 | 收藏、取消收藏、收藏列表、禁止收藏自己的商品 |
| 私信 | 创建会话、发送消息、引用、已读、删除、撤回 |
| 通知 | 预定、取消、完成通知和未读统计 |
| 异常场景 | 越权、重复预定、非法请求、已删除商品、跨会话引用 |
| 性能压测 | 商品列表、搜索、详情、并发预定 |
| 验收流程 | 从登录到完成交易的端到端业务流 |

## 4. 测试环境

| 类型 | 配置 |
|---|---|
| 后端语言 | Java 17 |
| 后端框架 | Spring Boot、Spring Security、Spring Data JPA |
| 测试框架 | JUnit 5、Spring Boot Test、MockMvc |
| 测试数据库 | H2 内存数据库，`application-test.yml` |
| 前端框架 | Vue 3、Vite、Vant |
| 构建验证 | `npm run build` |
| 压测工具 | `scripts/qa-load-test.js`，Node 原生 `fetch` |
| 统一响应 | 成功 `code=0`，业务失败通过 `code=400/403/404` 等表达 |

## 5. 测试策略

| 测试类型 | 策略 |
|---|---|
| 单元/组件级测试 | 针对 Service 和核心业务规则，通过 MockMvc 间接验证事务和校验逻辑 |
| 集成测试 | 使用 Spring Boot 上下文、H2 数据库和真实 Controller/Service/Repository 链路 |
| 异常场景测试 | 构造非法请求、越权用户、已下架商品、重复状态流转 |
| 并发测试 | 多线程同时请求预定接口，验证一物不能多卖 |
| 性能测试 | 在本地启动后端后，对热点接口进行并发请求，统计成功率和 P95 |
| 验收测试 | 以真实用户流程串联前后端功能 |
| 回归测试 | 每次合并前执行后端测试和前端构建 |

## 7. 执行命令

后端自动化测试：

```bash
cd backend
./mvnw test
```

前端构建验证：

```bash
cd frontend
npm run build
```

性能压测，需先启动后端：

```bash
cd backend
./mvnw spring-boot:run
```

另开终端执行：

```bash
node scripts/qa-load-test.js
```

可选参数：

```bash
BASE_URL=http://localhost:8080 TOTAL=200 CONCURRENCY=20 node scripts/qa-load-test.js
```

## 8. 进入与退出标准

### 8.1 进入标准

- 后端可以启动，测试 Profile 配置可用。
- 前端依赖已安装，`package-lock.json` 与 `package.json` 一致。
- 核心接口路径稳定，不再频繁变更。
- 测试数据可通过接口或初始化逻辑自动构造。

### 8.2 退出标准

- 后端自动化测试全部通过。
- 前端生产构建成功。
- 关键验收链路可以手工走通。
- 性能压测无大量 500 错误，核心接口成功率满足演示要求。
- 未实现能力已在测试报告中明确记录。

## 9. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| H2 与 MySQL 锁行为存在差异 | 并发预定在真实部署中表现可能不同 | 部署 MySQL 后补充同样并发脚本验证 |
| 前端缺少自动化 E2E | 视觉和浏览器兼容问题可能依赖人工发现 | 后续可引入 Playwright |
| 举报和 TTL 未实现 | 需求闭环存在扩展项 | 在验收报告中标记为后续迭代 |
| 开发期 Header 误用于正式环境 | 身份安全风险 | `mysql` profile 关闭 `dev-auth-enabled`，测试已覆盖关闭场景 |
