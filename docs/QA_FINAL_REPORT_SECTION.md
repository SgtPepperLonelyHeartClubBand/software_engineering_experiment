# 最终报告补充章节：测试与 QA 负责部分

> 本章节用于并入团队最终报告，覆盖测试与 QA 成员负责的《验收与发布》《运行与维护》内容。本文按当前仓库实际实现编写，后端为 Spring Boot，前端为 Vue/Vite。

## 一、测试目标与范围

本项目是面向校内师生的校园二手交易 Web 平台，测试工作的核心目标是验证系统是否能可靠支撑“发布、搜索、沟通、预定、线下交收、完成交易”的业务闭环。

测试范围包括：

| 模块 | 测试重点 |
|---|---|
| 认证与资料 | 一卡通号登录、JWT、开发期 Header、资料完善 |
| 商品 | 发布、浏览、搜索、详情、编辑、下架、图片上传 |
| 交易 | 预定、取消、完成、订单列表 |
| 状态机 | 在售、被预定、已完成、已下架 |
| 收藏与私信 | 收藏、会话、消息、引用、撤回、未读 |
| 通知 | 预定通知、取消通知、完成通知 |
| 异常场景 | 非法输入、越权、重复预定、已下架商品 |
| 并发场景 | 多买家同时预定同一商品 |
| 性能场景 | 商品列表、搜索、详情、预定接口压测 |

## 三、核心测试用例

### 1. 主流程验收

主流程如下：

```text
用户登录
  ↓
完善资料
  ↓
卖家发布商品
  ↓
买家搜索商品
  ↓
买家查看详情
  ↓
买家私信卖家
  ↓
买家点击“我想要”
  ↓
商品变为“被预定”
  ↓
卖家确认完成
  ↓
商品变为“已完成”
```

验收标准为：流程中每一步均有明确页面反馈或接口响应，状态变化能在商品详情、个人中心和通知中体现。

### 2. 状态机测试

| 场景 | 期望 |
|---|---|
| 在售商品被买家预定 | 商品变为“被预定”，订单为 `RESERVED` |
| 已预定商品再次被预定 | 返回 `code=400` |
| 买家取消预定 | 订单为 `CANCELLED`，商品恢复“在售” |
| 卖家确认完成 | 订单为 `COMPLETED`，商品变为“已完成” |
| 已完成订单再次取消 | 返回 `code=400` |
| 已下架商品被预定 | 返回 `code=404` |

### 3. 权限测试

| 场景 | 期望 |
|---|---|
| 未登录访问受保护接口 | HTTP 401 |
| 非发布者修改商品 | `code=403` |
| 卖家预定自己商品 | `code=400` |
| 买家确认交易完成 | `code=403` |
| 路人取消他人订单 | `code=403` |
| 非会话参与者查看消息 | `code=403` |

### 4. 异常输入测试

| 场景 | 期望 |
|---|---|
| 商品缺少图片 | `code=400` |
| 商品价格小于 0.01 | `code=400` |
| 商品标题超过 40 字 | `code=400` |
| 消息内容为空 | `code=400` |
| 消息超过 200 字 | `code=400` |
| 跨会话引用消息 | `code=400` |

### 5. 并发预定测试

并发测试模拟 6 个买家同时点击同一商品的“我想要”。测试结果要求：

- 只有 1 个请求返回 `code=0`；
- 其他请求返回 `code=400`；
- 商品最终状态为“被预定”；
- 商品想要人数只增加一次。

该用例验证了系统对“一物多卖”问题的防护能力。

## 四、性能压测

QA 新增性能压测脚本：

```text
scripts/qa-load-test.js
```

执行命令：

```bash
node scripts/qa-load-test.js
```

压测场景：

| 场景 | 接口 | 指标 |
|---|---|---|
| 商品列表 | `GET /api/items` | 成功率、平均耗时、P95 |
| 商品搜索 | `GET /api/items?keyword=QA` | 成功率、平均耗时、P95 |
| 商品详情 | `GET /api/items/{id}` | 成功率、平均耗时、P95 |
| 并发预定 | `POST /api/items/{id}/reserve` | 只有一个买家成功 |

该脚本用于课程演示阶段快速验证后端接口在并发请求下的稳定性。

## 五、验收结论

在当前实现范围内，系统已经完成校园二手交易平台的核心能力：用户可以登录、发布商品、浏览搜索、查看详情、收藏、私信、预定商品并完成交易。后端通过权限校验、参数校验、统一异常响应、事务和悲观锁保障交易状态一致性。

本轮测试执行结果如下：

| 项目 | 结果 |
|---|---|
| 后端自动化测试 | 通过，`Tests run: 15, Failures: 0, Errors: 0, Skipped: 0` |
| 前端构建验证 | 通过，生产构建产物已生成 |
| 性能压测 | 列表、搜索、详情接口成功率 100%；并发预定仅 1 个请求成功，其余请求返回业务拒绝，符合预期 |

经 QA 测试设计，系统具备基础交付条件。当前仍有两个方向属于后续扩展：

| 后续扩展项 | 说明 |
|---|---|
| 举报审核和管理员治理 | 本版本暂作为后续平台治理扩展能力 |
| 预定 TTL 自动释放 | 本版本暂作为后续可靠性增强能力 |

上述内容已记录为后续维护和迭代测试项，不影响当前版本完成基础校园二手交易闭环演示。

## 六、运行与维护

后端启动：

```bash
cd backend
./mvnw spring-boot:run
```

前端启动：

```bash
cd frontend
npm run dev
```

回归测试：

```bash
cd backend
./mvnw test
```

```bash
cd frontend
npm run build
```

维护建议：

| 方向 | 建议 |
|---|---|
| 安全 | 正式环境关闭开发期 Header，替换 JWT 密钥 |
| 数据库 | MySQL 部署后重新执行并发预定压测 |
| 性能 | 商品列表增加分页和索引，图片增加压缩 |
| 测试 | 后续引入 Playwright 做浏览器端自动化验收 |
| 业务 | 补充举报审核、管理员处理和预定 TTL |
